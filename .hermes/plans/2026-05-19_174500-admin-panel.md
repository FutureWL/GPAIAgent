# 后台管理员入口规划

## 目标
为 GPAIAgent 平台搭建独立的后台管理 Admin Panel，实现对平台所有数据的深度管理。

## 当前系统状态

**认证系统：** `JwtCookieAuthGuard`（JWT cookie），所有登录用户权限相同，无角色区分。
**User 模型：** 无 `role` 字段，所有用户平等。
**现有模块：** stocks, strategies, posts, stock-screen, ai, membership, market-sync, backtests, comments, auth

## 方案设计

### 1. 数据层 — 添加 Admin 角色体系

**修改 `schema.prisma`：**
```prisma
enum UserRole {
  USER   // 普通用户
  ADMIN  // 管理员
}

model User {
  role     UserRole  @default(USER)
  // ...
}
```

**数据库迁移：**
```bash
cd apps/api && npx prisma migrate dev --name add_user_role
```

### 2. 认证层 — Admin 权限守卫

**新建 `apps/api/src/auth/admin.guard.ts`：**
- 继承 `JwtCookieAuthGuard`
- 额外校验 `req.user.role === 'ADMIN'`
- 非 admin 返回 403 Forbidden

**保护策略：** 在需要 admin 权限的 controller 方法上叠加 `@UseGuards(JwtCookieAuthGuard, AdminGuard)`。

### 3. API 层 — Admin 模块

新建 `apps/api/src/admin/` 模块，提供以下管理接口：

| 端点 | 方法 | 功能 |
|------|------|------|
| `/admin/users` | GET | 分页查询用户列表 |
| `/admin/users/:id/role` | PATCH | 修改用户角色 |
| `/admin/users/:id/disable` | POST | 禁用/启用用户 |
| `/admin/stocks` | GET | 分页查询股票列表 |
| `/admin/stocks/sync` | POST | 手动触发全量数据同步 |
| `/admin/sync-queue` | GET | 查看同步队列状态 |
| `/admin/sync-jobs` | GET | 查看同步任务历史 |
| `/admin/posts` | GET | 管理帖子（删除/下架） |
| `/admin/strategies` | GET | 管理策略（审核/下架） |
| `/admin/stats` | GET | 平台统计（用户数/股票数/帖子数等） |

### 4. 前端层 — Admin Panel 页面

**路由：** `/admin`（独立于主站的布局）

**目录结构：**
```
apps/web/src/app/admin/
├── layout.tsx          # Admin 专属布局（侧边导航）
├── page.tsx            # Dashboard 总览
├── users/
│   └── page.tsx         # 用户管理
├── stocks/
│   └── page.tsx         # 股票数据管理
├── sync/
│   └── page.tsx         # 同步队列监控
├── posts/
│   └── page.tsx         # 内容管理
└── strategies/
    └── page.tsx          # 策略管理
```

**入口保护：** `/admin` 路由在客户端检查用户 role，非 admin 重定向至首页。

## 内容审核流程

帖子/策略发布后状态 `status` = `pending_review`，管理员可：
- **通过** → `status` = `published`
- **拒绝** → `status` = `rejected`（附拒绝原因）
- **下架** → `status` = `removed`（已发布的也可以下架）

拒绝/下架时需填写原因，存入 `rejectReason` 字段。

## 操作日志（审计日志）

**新建 `AuditLog` model：**
```prisma
model AuditLog {
  id         String   @id @default(cuid())
  adminId    String              // 执行操作的管理员
  action     String              // 操作类型：USER_ROLE_CHANGE / POST_REVIEW / STRATEGY_REVIEW / STOCK_SYNC / etc.
  targetType String              // 目标类型：User / Post / Strategy / Stock / SyncJob
  targetId   String              // 目标ID
  detail     Json?               // 变更详情（before/after）
  ip         String?             // IP地址
  createdAt  DateTime @default(now())
}
```

## 风险与tradeoff

- **迁移风险：** User 表加字段是 DDL 操作，数据量小无风险。
- **权限漏洞：** 所有管理接口必须手动加 `@UseGuards`，遗漏则普通用户可访问，需代码审查覆盖。
- **Seed 第一个 admin：** 必须通过 seed 创建首个 admin 账号。
- **审核状态展示：** 主站用户查询帖子/策略时需过滤 `pending_review` 状态（管理员审核中不对用户可见）。
